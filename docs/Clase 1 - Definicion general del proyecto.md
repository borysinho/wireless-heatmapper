# Perfil de Proyecto – Apuntes de Clase

## Primer Parcial
**Entrega:** 21 al 28 de abril  
**Sprint 1:** 3 semanas

---

# Estructura del Documento

1. Carátula  
2. Tabla de contenidos  
3. Introducción  
4. Antecedentes  
5. Descripción del problema  
6. Situación problemática  
7. Situación deseada  
8. Objetivos  
9. Alcance  
10. Tecnología de desarrollo  
11. Innovación  
12. Bibliografía  
13. Anexos  

---

# 1. Carátula

La carátula debe incluir:

- UAGRM
- FICCT
- Carrera
- Proyecto (título completo)
- Materia
- Grupo #
- Apellidos y nombres completos
- Registro
- Lugar y fecha

**Nota:**  
- Solo utilizar logo y nombre de la carrera o universidad.

---

# 2. Tabla de Contenidos o Índice

- Generado para facilitar la navegación.
- Debe tener al menos dos niveles.

---

# 3. Introducción

La introducción debe:

- Evidenciar que el contenido es inédito y elaborado por el grupo.
- Explicar claramente el tema.
- Motivar al lector para que continúe leyendo el documento.

---

# 4. Antecedentes

## Recomendaciones

- Hacer referencia a situaciones o proyectos anteriores.
- Incluir fundamentación teórica.
- Realizar una breve descripción relacionada al proyecto.

## Comparación de software

- Comparar aproximadamente 3 aplicaciones o proyectos similares.
- Analizar aspectos relevantes de cada uno.

---

# 5. Descripción del Problema

## Contenido

- Definir completamente el problema abordado.
- Explicar el contexto operativo donde se pondrá en marcha.
- Utilizar diagramas que ayuden a comprender el problema.

### Ejemplos de diagramas

- Ishikawa
- Diagramas de negocio

---

# 6. Situación Problemática

## Recomendaciones

- No extenderse demasiado.
- Aproximadamente entre 5 y 10 líneas.
- Hablar únicamente del problema principal.

## Importante

- No hablar de software ni tecnología.
- Centrarse en el problema del caso de estudio.

---

# 7. Situación Deseada

## Recomendaciones

- Entre 5 y 10 líneas.
- Explicar cómo cambiaría el escenario del caso de estudio.
- Describir cómo mejoraría la situación una vez implementada la solución.

## Importante

- No enfocarse únicamente en el software.
- Explicar el impacto positivo esperado.

---

# 8. Objetivos

## Recomendaciones Generales

- Estar libres de ambigüedad.
- No ser demasiado extensos.
- Ser claros y precisos.
- Utilizar verbos de acción.

## Objetivo General

- Debe representar el objetivo principal del proyecto.

## Objetivos Específicos

- No hablar directamente de funcionalidades.
- Explicar qué se realizará para alcanzar el objetivo general.
- Cada objetivo debe tener un entregable.

### Ejemplos de entregables

- Código
- Modelo
- Diseño

---

# 9. Alcance

## Contenido

- Explicar qué hará el sistema.
- Definir funcionalidades principales.

## Organización sugerida

- Organizar por:
  - Épicas
  - Módulos
  - Paquetes

## Consideraciones

- Identificar requerimientos principales.
- Definir líneas estratégicas.
- Posible menú principal.
- Explicar cada función principal.

---

# 10. Tecnología de Desarrollo

## Debe incluir

- Lenguaje(s) de programación a utilizar.
- Justificación de elección de tecnologías.
- Herramientas para desarrollo colaborativo.
- Herramientas CASE.
- Tecnologías de implementación.

## Ejemplos

### Backend
- Framework
- API
- Base de datos

### Frontend
- Framework frontend

### Despliegue
- CI/CD
- Herramientas de despliegue

## Proceso de Desarrollo

- Explicar qué proceso de desarrollo se utilizará.
- Describir cómo se llevará a cabo el desarrollo del software.

---

# 11. Bibliografía

## Referencias

Las referencias deben estar relacionadas con las bases utilizadas para desarrollar el proyecto.

## Fuentes recomendadas

- Libros físicos o digitales
- Sitios web especializados
- Entrevistas
- Documentación técnica

## Formato

- Utilizar normas APA.

---

# 12. Anexos

## 12.1 Esquema gráfico

Puede incluir:

- Situación problemática
- Situación deseada

## 12.2 Datos del caso de estudio

- Información relevante del caso analizado.

## 12.3 Currículum Vitae

- Información de integrantes o responsables.

---

# Recomendaciones Finales

- Mantener redacción clara y formal.
- Evitar ambigüedades.
- Utilizar verbos de acción en objetivos.
- Organizar adecuadamente los módulos y funcionalidades.
- Mantener coherencia entre problema, objetivos y alcance.
